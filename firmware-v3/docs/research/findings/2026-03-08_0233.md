## Finding 1: Autorubric — Production-Grade Rubric Execution for LLM-as-Judge

**Paper:** Autorubric: A Unified Framework for Rubric-Based LLM Evaluation
**arXiv:** [2603.00077](https://arxiv.org/abs/2603.00077) (February 2026)
**Authors:** Delip Rao, Chris Callison-Burch

**Relevance:** Provides the *execution layer* for rubric-based MLLM evaluation — complementing RRD (arXiv:2602.05125) which tells you how to *design* rubrics. Together, these two papers form a complete rubric design→execution pipeline for our L4 aesthetic judge.

**Key transferable methods:**

1. **Position bias mitigation via option shuffling** — when presenting ordinal quality levels (1-5) to the MLLM judge, shuffle the order across evaluations to eliminate positional preference. Directly applicable to our aesthetic scoring prompts.

2. **Verbosity bias via length penalties** — MLLM judges tend to score outputs higher when the evaluation prompt or context is longer. Apply length normalisation to prevent our capture descriptions from influencing scores based on verbosity rather than quality.

3. **Per-criterion atomic evaluation** — evaluate each rubric criterion independently with its own prompt/response cycle, preventing criterion conflation (where scoring "colour harmony" bleeds into "temporal smoothness" scoring). This maps directly to our layered architecture: evaluate each L4 sub-criterion atomically.

4. **Multi-judge ensemble with configurable aggregation** — run the same evaluation prompt through multiple MLLM instances (or different models) and aggregate via majority vote, weighted vote, or unanimous agreement. For our system: run 3 evaluation passes per capture and take weighted majority as the L4 score.

5. **Psychometric reliability metrics** — includes Cohen's κ for inter-judge agreement. We can use this to monitor MLLM judge consistency over time and detect drift.

**How we'd use it:** Apply all five techniques to our L4 MLLM-as-judge pipeline:
- Design rubrics with RRD's recursive decomposition
- Execute evaluations with Autorubric's bias mitigations
- Aggregate with multi-judge ensemble
- Monitor with psychometric reliability metrics

The open-source Python framework can be integrated directly.

**Implementation complexity:** Low — open-source Python library, primarily prompt engineering and evaluation pipeline orchestration. No model training required.

**Layer affected:** L4 (aesthetic evaluation) — execution reliability

## Finding 2: LLM-AMARL — LLM-Empowered Multi-Agent RL for Dynamic Lighting Control

**Paper:** Personalized Dynamic Lighting via LLM-Empowered Multi-Agent Reinforcement Learning: From User Intent to Photometric Control
**Source:** Expert Systems with Applications (ScienceDirect), published ~February 2026
**Semantic Scholar:** [aefcd9810080d8fcb2301a42bb5f198d65f138d1](https://www.semanticscholar.org/paper/aefcd9810080d8fcb2301a42bb5f198d65f138d1)

**Relevance:** The closest published system to our closed-loop LED optimisation architecture — uses LLM for high-level intent interpretation combined with MARL for low-level multi-fixture photometric actuation. This is stage lighting automation with RL, which is Domain 8 in our research targets.

**Key transferable methods:**

1. **Hierarchical LLM→MARL architecture** — LLM parses high-level intent (e.g., "avoid glare during reading") into structured photometric targets, then MARL agents control individual fixtures to achieve those targets. For our system: LLM analyses audio context/mood → structured visualisation targets → RL agent tunes per-LED-segment parameters.

2. **DBSCAN density clustering with composite distance metric** — clusters user preferences using a composite metric fusing illuminance, colour temperature, and time. The exponential decay-based dynamic weighting captures time-varying preferences. For our system: cluster "good" visualisation parameter sets by audio context using a composite metric fusing spectral centroid, energy, beat density. This builds a preference model — "what works for EDM drops" vs "what works for ambient intros."

3. **CIE-compliant photometric evaluation** — evaluates against CIE 175-2005 and CIE 121-1994 standards using high-fidelity simulation. While our LED strip isn't held to CIE indoor lighting standards, the principle of calibrating against a known perceptual standard (rather than ad-hoc metrics) is directly transferable.

**How we'd use it:** The architectural pattern validates and extends our planned closed-loop system:
- Audio analysis (existing) → LLM context extraction (new) → structured parameter targets (new) → RL/BO optimiser (planned) → LED parameter adjustment (existing)
- DBSCAN preference clustering provides a mechanism to build audio-context-aware parameter presets automatically, rather than hand-tuning per genre/mood

**Implementation complexity:** High — full MARL training is compute-intensive. However, the architectural pattern (LLM intent → optimizer actuation) and the DBSCAN preference clustering are independently implementable at lower cost.

**Layer affected:** Optimisation (closed-loop strategy) + Pipeline (audio-to-intent bridge)
