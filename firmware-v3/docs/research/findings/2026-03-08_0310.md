## 🔴 HIGH-VALUE FINDING

### DecDTW — Exact Differentiable DTW via Implicit Differentiation
**arXiv:** [2303.10778](https://arxiv.org/abs/2303.10778) (ICLR 2023)

**Relevance:** Provides the exact differentiable alignment loss needed to make our L2 audio-visual sync metric end-to-end trainable. Deep Time Warping (previous sweep) handles multi-series alignment but relies on learned decomposition; DecDTW provides the mathematical foundation for computing exact gradients through the optimal warping path.

**Key transferable method:** Formulates DTW as a continuous, inequality-constrained optimisation problem and uses implicit differentiation (deep declarative networks) to compute gradients of the optimal warping path. Unlike Soft-DTW (which blurs alignment boundaries through temperature-softened min operators), DecDTW recovers the **exact** optimal path and differentiates through it.

**Critical advantage over Soft-DTW:** Soft-DTW gradients point toward "approximately aligned" — they spread probability mass across nearby alignment candidates. DecDTW gradients point toward "exactly aligned" — they tell the optimiser precisely which LED frames need to shift to match which audio features. For our system, where we care about tight beat-to-visual synchronisation (not approximate), this precision matters.

**How we'd use it:** Define the L2 alignment loss as:
```
L2_loss = DecDTW(LED_brightness_series, audio_onset_strength)
        + DecDTW(LED_colour_change_series, spectral_centroid_trajectory)
```
Gradients flow from this loss through the entire parameter space, telling the optimiser: "shift this LED transition 12ms earlier to align with the beat." The paper demonstrates this on music audio-to-score alignment — our LED-to-audio alignment is structurally identical.

**Implementation complexity:** Medium — the implicit differentiation framework is published and well-documented. Main work: adapting the continuous DTW formulation to our multi-channel time series (320 LED channels × audio features).

**Layer affected:** L2 (audio-visual alignment) — exact differentiable alignment loss

## 🔴 HIGH-VALUE FINDING

### Evaluator Stress Tests (EST) — Detecting Proxy Gaming in RL
**arXiv:** [2507.05619](https://arxiv.org/abs/2507.05619) (July 2025, revised January 2026)

**Relevance:** Once our closed-loop optimiser is running, the single most dangerous failure mode is **reward hacking** — the optimiser finds LED parameter combinations that score well on L0-L4 metrics without actually producing good visualisations. EST provides a systematic method to detect and mitigate this.

**Key transferable methods:**

1. **Controlled perturbation with semantic validity audit** — apply small, semantically meaningless changes to the LED output (e.g., shift all colours by 1 LSB, add 1ms latency to all transitions). If the quality metrics change significantly, they are exploitably sensitive to noise rather than measuring genuine quality. EST separates "exploitable sensitivity" from "content-driven improvements."

2. **Invariance-based gaming detection** — track which metric dimensions the optimiser preferentially exploits. If the optimiser consistently increases L4 aesthetic scores by making imperceptible changes (rather than visible improvements), it's gaming L4. EST achieves 78.4% precision / 81.7% recall at detecting gaming episodes.

3. **Closed-loop mitigation** — once gaming is detected, reduce the weight of the exploited metric and increase exploration. Reduces reward hacking by 54.6% in RL environments.

**How we'd use it:** Run EST periodically on the optimiser's output:
- Perturb LED parameters by ε in random directions
- Measure L0-L4 metric responses
- If any metric changes disproportionately to the perturbation magnitude → flag as exploitable
- Reduce that metric's weight in the reward, increase curiosity/exploration bonus

This is the "immune system" for our closed-loop system. Without it, we'll inevitably converge on reward-hacking patterns.

**Implementation complexity:** Low-Medium — the perturbation framework is straightforward. The statistical tests (invariance analysis) are standard. Main work: defining "semantically meaningless" perturbations for LED parameters (which is domain-specific).

**Layer affected:** Optimisation (reward robustness / gaming prevention)

## Finding 3: PAR (Preference As Reward) — Bounded Reward Shaping Principles
**arXiv:** [2502.18770](https://arxiv.org/abs/2502.18770) (February 2025, revised January 2026)

**Relevance:** Provides two concrete design principles for shaping our L0-L4 reward signals to prevent reward hacking, plus a specific mechanism (preference extraction) that's more robust than raw metric scores.

**Key transferable methods:**

1. **Bounded rewards** — cap all metric outputs. Our L0-L4 metrics should have defined upper bounds. Unbounded rewards (e.g., "pixel similarity score can grow arbitrarily") create infinite-gain exploitation opportunities. Apply sigmoid/tanh bounding to each metric.

2. **Rapid initial growth → gradual convergence** — shape the reward curve so early improvements (from terrible to acceptable) give large reward gains, but later improvements (from good to great) give diminishing returns. This prevents the optimiser from spending all its budget chasing marginal metric improvements instead of exploring.

3. **Preference As Reward (PAR)** — instead of feeding raw metric scores (L0=0.73, L2=0.85, L4=3.2) to the optimiser, extract the **preference ordering** embedded in the metrics: "Visualisation A is better than B because L2 improved while L4 held steady." The preference signal is more robust than raw scores because it's invariant to metric scale, calibration drift, and absolute value — it only cares about relative ordering.

**How we'd use it:** Transform our metric pipeline output:
```
Raw: {L0: 0.73, L1: 0.91, L2: 0.62, L3: 0.78, L4: 3.1}
→ Bounded: sigmoid(L0), sigmoid(L1), ...
→ Shaped: rapid_growth(bounded_score)
→ PAR: compare(current, previous) → preference_signal
```

The optimiser receives the preference signal, not raw scores. This makes it robust to metric miscalibration and harder to game.

**Implementation complexity:** Low — the shaping functions are simple transforms. PAR requires storing previous evaluations for comparison, which is trivial.

**Layer affected:** Optimisation (reward signal design)

## Finding 4: CD-RLHF — Curiosity-Driven Diversity in Optimisation
**arXiv:** [2501.11463](https://arxiv.org/abs/2501.11463) (January 2025, ACL 2025)

**Relevance:** Prevents the optimiser from converging on a single "safe" visualisation pattern (mode collapse). Without diversity pressure, the optimiser will find one colour palette and transition speed that scores well on L0-L4 and never deviate — producing monotonous visualisations that are technically high-quality but aesthetically boring.

**Key transferable method:** Intrinsic curiosity reward added alongside the extrinsic (L0-L4) reward. The curiosity signal encourages the optimiser to explore parameter regions it hasn't visited before. The key result: diversity improves on multiple metrics while alignment quality remains comparable to standard RLHF.

**How we'd use it:** Add a curiosity bonus to the reward:
```
total_reward = extrinsic_reward(L0-L4) + λ * curiosity_reward(parameter_novelty)
```

Where `curiosity_reward` measures how different the current LED parameter configuration is from recently explored configurations (e.g., L2 distance in parameter space from a rolling buffer of recent parameter sets). This pushes the optimiser to try different colour palettes, decay rates, pattern types, and transition speeds — even when its current configuration scores well.

**Critical for our domain:** Music visualisation demands variety. A technically perfect but repetitive visualisation is worse than a slightly imperfect but varied one. Curiosity-driven exploration is the mechanism that produces variety.

**Implementation complexity:** Low — curiosity reward is a distance computation in parameter space. The balance parameter λ requires tuning but has well-studied heuristics.

**Layer affected:** Optimisation (exploration/diversity)

## Finding 5: Reward Distillation — Fast Proxy from Expensive Evaluator Traces
**arXiv:** [2502.19557](https://arxiv.org/abs/2502.19557) (February 2025)

**Relevance:** Solves the latency bottleneck identified in MSVBench: our MLLM-as-judge (L4) is too slow for real-time evaluation in a 120 FPS loop. Reward distillation trains a fast, lightweight model that approximates the expensive evaluator's judgments.

**Key transferable method:** Generate **pseudo-rewards** from the structure of teacher responses without needing the teacher to provide explicit reward signals. The distilled student model can surpass teacher performance (demonstrated on GSM8K, MMLU-PRO). The key insight: the reward information is implicit in the teacher's output distribution, not just in its final scores.

**How we'd use it:**
1. Run the MLLM-as-judge on N captured LED visualisations (offline, expensive, ~seconds per evaluation)
2. Collect the MLLM's reasoning traces, not just final scores
3. Train a small MLP or CNN to predict the MLLM's scores from raw LED frame features
4. Deploy the distilled model in the real-time evaluation loop (~milliseconds per evaluation)
5. Periodically re-run the MLLM teacher to check for drift and retrain the student

This gives us MLLM-quality aesthetic evaluation at CNN inference speeds. The student doesn't need to understand "why" a visualisation is good — it just needs to predict the MLLM's score from the same features.

**Implementation complexity:** Medium — requires collecting a dataset of MLLM evaluations (hundreds to thousands), training a small model, and building the periodic recalibration pipeline.

**Layer affected:** L4 (aesthetic evaluation — fast proxy) + Pipeline (real-time evaluation architecture)

## Design Principle: Luminance-Priority Transitions for Perceived Smoothness

**Source:** Psychophysics literature on apparent motion perception (multiple sources, not a single new paper)

**Key finding:** Luminance contrast is substantially more effective than chromatic contrast for producing perceived smooth apparent motion. The human visual system detects motion primarily through the magnocellular (luminance) pathway, which has higher temporal resolution (~60+ Hz) than the parvocellular (chromatic) pathway (~15-25 Hz).

**Implication for our system:** LED animations should prioritise luminance-domain transitions (brightness changes) for temporal coherence and perceived smoothness. Chromatic transitions (colour hue shifts) can be slower because the visual system processes them at lower temporal resolution. Specifically:
- Beat-synchronised transitions: use brightness pulses (perceived as crisp and responsive)
- Mood/section transitions: use colour hue shifts (can be gradual, still perceived as smooth)
- Avoid rapid chromatic-only transitions — they'll appear to stutter even at 120 FPS because the parvocellular pathway can't track them

**Not a paper finding but a validated design principle.** Relevant to: L0 (frame design), L1 (temporal coherence evaluation — weight luminance transitions higher), L4 (aesthetic rubric — penalise rapid chromatic-only transitions).

## Identified Research Gap: Differentiable LED Strip Rendering

**No papers found.** This is an opportunity, not a finding.

Our LED strip is a trivially simple renderer: 320 × (R,G,B) values → photon emission. Unlike 3D scene rendering (where differentiable rendering is a major research area), our "scene" has no geometry, no lighting, no occlusion. The entire rendering pipeline is:

```
parameters → pattern_generator(params, audio_features) → RGB[320] → WS2812_gamma_correction → photons
```

Every step is differentiable (or can be made so). If we also model the camera capture:

```
photons → camera_response(exposure, white_balance) → captured_image → VDP_quality_score
```

Then the entire chain from parameters to quality score is differentiable. This enables direct gradient-based optimisation: "what parameter change maximally improves perceived quality?"

No existing paper addresses this because LED strip rendering is too simple for the computer graphics community and too hardware-specific for the ML community. We would need to build this ourselves — but the components exist:
- Pattern generator: our existing firmware (make differentiable via PyTorch surrogate)
- Gamma correction: known polynomial, trivially differentiable
- Camera model: CameraVDP provides this
- Quality score: ColorVideoVDP provides differentiable quality prediction

**This is the end-to-end gradient path** that connects all our findings into a single optimisation framework.
