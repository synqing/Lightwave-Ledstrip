## Finding 1: RESCUE — Causal Multi-Fidelity Bayesian Optimisation for Budget-Optimal Metric Layer Selection

**Paper:** arXiv:2602.00788 — "Multi-Objective Multi-Fidelity Bayesian Optimization with Causal Priors" (January 31, 2026)
**Authors:** Md Abir Hossen, Mohammad Ali Javidian, Vignesh Narayanan, Jason M. O'Kane, Pooyan Jamshidi

**One-sentence relevance:** Provides a principled framework (RESCUE) for deciding WHEN to run expensive evaluation layers (L3 perceptual, L4 aesthetic) vs. cheap ones (L0 frame, L1 temporal) during our closed-loop optimisation, using causal models of how fidelity choices affect quality objectives rather than naive scheduling.

**Key transferable method:**

1. **Causal Structural Model for Fidelity Selection:** RESCUE learns a structural causal model (SCM) that captures intervention effects between inputs (parameter configurations), fidelities (metric layers), and objectives (quality scores). The critical distinction from standard multi-fidelity BO: conventional approaches model *associational* dependencies between fidelities (e.g., "L0 and L2 are correlated") whereas RESCUE models *interventional* dependencies (e.g., "running L2 on configuration X after observing L0=0.3 will yield Y with probability Z"). This means the optimiser understands the causal mechanism of how cheap evaluations predict expensive ones, not just their statistical correlation.

2. **Causal Hypervolume Knowledge-Gradient (CHKG) Acquisition:** The acquisition function selects input-fidelity pairs that maximise expected multi-objective hypervolume improvement per unit cost. In our context: given a candidate parameter configuration, CHKG decides whether to evaluate it with just L0+L1 (cheap, ~1ms), add L2 audio-visual alignment (~10ms), or commit to full L3+L4 evaluation (~100ms+). The causal model ensures this decision accounts for how lower-fidelity results causally influence higher-fidelity outcomes.

3. **Multi-Objective Simultaneous Optimisation:** RESCUE handles multiple competing objectives natively. Our system has at minimum 5 quality dimensions (L0–L4) that partially conflict (e.g., maximising L2 beat-synchronisation often degrades L1 temporal smoothness through abrupt transitions). RESCUE constructs Pareto fronts across these objectives while simultaneously deciding which fidelity level to evaluate.

**Concrete application to our system:**

- **Adaptive Metric Pipeline Depth:** Currently, the metric pipeline concept runs all layers for every evaluation. RESCUE provides a principled stopping rule: after L0+L1 evaluation, the causal model predicts whether higher layers are likely to produce useful gradient signal for this configuration. If L0=0.95 and L1=0.85, the causal model may predict L3+L4 will also be high → skip expensive evaluation, use predicted scores. If L0=0.6 and L1=0.4, run full pipeline because the configuration needs detailed diagnosis. Expected cost reduction: 60-80% of evaluation budget on configurations where cheap metrics are predictive.

- **Interaction with REACT FMG-DFS (documented 2026-03-09_2359):** REACT's frame-sampling selects WHICH frames to evaluate. RESCUE selects WHICH METRIC LAYERS to run on those frames. These compose: FMG-DFS reduces frame count → RESCUE reduces per-frame metric depth → multiplicative evaluation cost reduction.

- **Interaction with multiplicative composition (documented 2026-03-10_0110):** ToolRLA's multiplicative reward composition provides the multi-objective quality function. RESCUE's causal BO provides the search strategy over that quality function. These are complementary: quality function defines WHAT to optimise; RESCUE defines HOW to search efficiently.

**Implementation complexity:** Medium — requires building a causal graph over our metric layers (straightforward given our known L0→L1→L2→L3→L4 dependency structure), fitting GP surrogates per layer, and implementing the CHKG acquisition function. The causal graph structure is mostly known a priori (L0 predicts L1, L0+L1 predict L2, etc.), which significantly simplifies the SCM learning phase.

**Target layer:** Optimisation (closed-loop evaluation budget allocation) + Architecture (metric pipeline depth control)

## Finding 2: Omni-Judge — Tri-Modal LLM Evaluation Taxonomy and High-FPS Failure Mode for L4 Design

**Paper:** arXiv:2602.01623 — "Omni-Judge: Can Omni-LLMs Serve as Human-Aligned Judges for Text-Conditioned Audio-Video Generation?" (February 2, 2026)
**Authors:** Susan Liang, Chao Huang, Filippos Bellos, Yolo Yunlong Tang, Qianxiang Shen, Jing Bi, Luchuan Song, Zeliang Zhang, Jason Corso, Chenliang Xu

**One-sentence relevance:** Establishes a 9-dimension evaluation taxonomy for audio-video quality judging AND demonstrates that omni-LLMs critically underperform on high-FPS temporal metrics — providing both a structured rubric for our L4 MLLM-as-judge and a clear negative result that constrains L4's role in our pipeline.

**Key transferable method:**

1. **9-Dimension Perceptual Evaluation Taxonomy:** Omni-Judge decomposes audio-video quality into 9 perceptual and alignment dimensions, systematically evaluated by omni-LLMs (models that process audio+video+text jointly). The taxonomy covers: audio-text alignment, video-text alignment, audio-video-text coherence, video quality, audio quality, audio-video synchronization, temporal quality, perceptual realism, and semantic plausibility. This provides a structured rubric template for our L4 layer.

2. **High-FPS Temporal Resolution Failure (CRITICAL NEGATIVE FINDING):** The paper demonstrates that omni-LLMs "underperform on high-FPS perceptual metrics such as video quality and audio-video synchronization, due to limited temporal resolution." This is directly actionable: our system runs at 120 FPS. If omni-LLMs cannot reliably judge temporal quality at high frame rates, then L4 MLLM-as-judge MUST NOT be used for L1 (temporal coherence) or L2 (beat-level synchronisation) evaluation. These layers must use hard computational metrics (frame-difference-based for L1, cross-correlation-based for L2). The MLLM should be confined to L3 (perceptual similarity at macro level) and L4 (aesthetic evaluation at segment level).

3. **Interpretable Chain-of-Thought Feedback for Refinement:** Omni-Judge produces "interpretable explanations that expose semantic or physical inconsistencies, enabling practical downstream uses such as feedback-based refinement." For our PROPOSE→APPLY→VERIFY loop: the L4 MLLM evaluation should output not just a score but structured chain-of-thought explaining WHAT is wrong ("the colour transitions during the chorus are too abrupt", "the bass-driven sections lack warm tones"). This structured feedback becomes the input to the PROPOSE step, constraining what parameter changes to explore next.

4. **Failure of Isolated Modality-Pair Metrics:** The paper shows that traditional metrics (FVD, CLAP, ViCLIP) "focus on isolated modality pairs, struggle with complex prompts, and provide limited interpretability." This validates our multi-layer approach — single metrics operating on one modality pair (audio-only or visual-only) are insufficient. The 5-layer pipeline with cross-modal metrics (L2) is architecturally sound.

**Concrete application to our system:**

- **L4 Scope Constraint (DESIGN DECISION):** Omni-Judge's negative finding establishes a hard design rule: L4 MLLM-as-judge operates ONLY on temporally downsampled representations (e.g., 1-second composite snapshots, not individual frames). All sub-second temporal quality evaluation must use L0/L1 hard metrics. This prevents wasting MLLM calls on a task where they demonstrably fail.

- **Structured L4 Evaluation Rubric:** Adapt Omni-Judge's 9-dimension taxonomy into our L4 prompt template. For LED strip visualisation, the relevant dimensions map to: visual quality (colour accuracy, brightness range), audio-visual synchronisation (at macro/segment level, not beat level), aesthetic coherence (colour palette harmony across a segment), perceptual realism (does the visualisation "feel" music-appropriate). Irrelevant dimensions (text alignment) are dropped.

- **CoT Feedback → PROPOSE Pipeline:** The interpretable output format feeds directly into parameter proposal. Instead of blind search, the optimiser receives structured feedback: "L4 says colour palette is too monotone during verse sections" → PROPOSE adjusts colour diversity parameters specifically for low-energy audio segments.

**Implementation complexity:** Low for the scope constraint and rubric adaptation (prompt engineering). Medium for the CoT-to-PROPOSE pipeline (requires parsing structured MLLM output into parameter search directions).

**Target layer:** L4 (aesthetic evaluation design) + Architecture (MLLM role boundaries) + Optimisation (feedback-guided proposal generation)
