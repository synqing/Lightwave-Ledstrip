## HIGH-VALUE FINDING

### T-LPIPS (Temporal LPIPS) — Perceptual Temporal Consistency Metric

- **Source:** PhysVideoGenerator (arXiv:2601.03665, Jan 7, 2026)
- **Relevance:** Directly fills a gap in our L1 temporal coherence layer — we currently use VBench temporal_flickering and WCS flicker penalty, but have no *perceptual* temporal consistency metric that accounts for how humans perceive frame-to-frame changes.
- **Key transferable method:** T-LPIPS computes the difference in perceptual quality change between consecutive frames: `|LPIPS(gen_t, gen_{t+1}) - LPIPS(ref_t, ref_{t+1})|`. Rather than checking if adjacent frames are similar (which penalises legitimate motion), it checks whether the *magnitude of perceptual change* between consecutive generated frames matches the expected magnitude from a reference pair. This enforces consistent transition dynamics without suppressing intended movement.
- **How we'd use it:** For our 320×1 LED strip at 120 FPS, compute LPIPS (or a lightweight proxy — e.g. VGG-16 feature L2 distance) between consecutive captured frames. Compare the temporal delta profile against the audio energy envelope's first derivative. A well-synchronised visualisation should have perceptual deltas that track audio energy changes — high T-LPIPS during beat transients is expected; high T-LPIPS during sustained tones indicates flickering.
- **Implementation complexity:** Medium — requires a perceptual feature extractor (LPIPS net or lighter VGG variant). The 1D 320-pixel signal may need reshaping to 2D for standard LPIPS models, OR we use a 1D feature extractor trained on LED frame statistics.
- **Target layer:** L1 (temporal coherence)
- **Significance:** This is the first metric we've found that distinguishes *intended motion* from *unwanted flicker* at a perceptual level. VBench temporal_flickering and WCS catch gross flickering but don't differentiate perceptual significance. T-LPIPS does.

## HIGH-VALUE FINDING

### RRD — Rubric Refinement through Decomposition for LLM/MLLM Judge Scoring

- **Paper:** "Rethinking Rubric Generation for Improving LLM Judge and Reward Modeling for Open-ended Tasks" (arXiv:2602.05125, Feb 4, 2026)
- **Authors:** Shen, Qiu, Whitehouse, Alazraki, Goel, Barbieri, Willi, Mathur, Leontiadis
- **Relevance:** Directly addresses our L4 aesthetic evaluation's biggest problem — high variance across MLLM-as-judge calls. RRD provides a systematic method to refine evaluation rubrics so they produce more consistent, discriminative scores.
- **Key transferable method:** Three-stage rubric refinement:
  1. **Recursive Decomposition** — takes coarse evaluation criteria (e.g. "visual quality") and decomposes into fine-grained, discriminative sub-criteria (e.g. "colour transitions are smooth", "brightness changes track beat onsets", "no more than 2 adjacent pixels share identical RGB values during movement")
  2. **Misalignment Filtering** — removes rubric items whose scoring direction doesn't correlate with overall quality (i.e. higher score on that criterion doesn't reliably predict higher overall quality)
  3. **Correlation-Aware Weighting** — prevents over-counting rubric items that measure nearly the same thing by detecting correlated criteria and down-weighting redundant ones
- **Quantitative results:** +17.7 points on JudgeBench across GPT-4o and Llama-405B judges. Reward model improvements of 60-160% over baseline rubrics. Transfers across benchmarks.
- **How we'd use it:** Apply RRD to our current aesthetic evaluation prompt. Take our existing rubric dimensions (colour harmony, motion quality, beat responsiveness, dynamic range, pattern variety) and run the decompose-filter-weight cycle to produce a refined rubric. This should measurably reduce scoring variance across repeated MLLM calls on the same frame sequence.
- **Implementation complexity:** Low — this is a prompt-engineering / rubric-design method, not a model change. The decomposition can be done once offline and the resulting rubric baked into our L4 evaluation prompts.
- **Target layer:** L4 (aesthetic evaluation)
- **Significance:** Our current MLLM-as-judge approach suffers from ~15-20% scoring variance across identical inputs. RRD provides a principled, tested method to reduce this. The correlation-aware weighting is particularly valuable — it prevents our rubric from double-counting related dimensions (e.g. "colour transitions" and "smooth hue changes" are highly correlated and should be weighted as one signal, not two).

## Finding 3

### VisionDirector — GRPO-Optimised VLM Planning for Closed-Loop Refinement

- **Paper:** "VisionDirector: Vision-Language Guided Closed-Loop Refinement for Generative Image Synthesis" (arXiv:2512.19243, Dec 2025)
- **Relevance:** Provides a tested architectural pattern for our PROPOSE→APPLY optimisation loop: VLM as planner → micro-edits → verify/rollback → GRPO to learn optimal trajectory policies.
- **Key transferable method:** The paper separates the *planning* component (which decides what to change) from the *rendering* component (which executes the change). The planner is a VLM that:
  1. Extracts structured goals from quality metrics
  2. Decides: STOP / VERIFY / EDIT for each goal
  3. Issues micro-edits (small parameter changes, not wholesale rewrites)
  4. Verifies each edit against the quality metric, rolls back if regression
  5. Is fine-tuned with Group Relative Policy Optimization (GRPO) — reward = alignment improvement, penalty = number of edit steps — yielding shorter, more efficient edit trajectories (3.1 vs 4.2 median steps, ~26% latency reduction)
- **How we'd use it:** Map this to our parameter optimisation:
  - **Planner** = LLM/MLLM that reads L0-L4 metric scores and proposes parameter adjustments
  - **Renderer** = ESP32 firmware executing the parameters
  - **Verify** = re-capture + re-evaluate metrics after parameter change
  - **GRPO training signal** = metric improvement per step, penalised by step count
  - The verify/rollback pattern prevents quality regressions during optimisation
  - GRPO could train a lightweight policy model that learns which parameter changes yield high metric improvement, reducing the number of iterations needed
- **Implementation complexity:** High — the full GRPO loop requires multiple rollouts. However, the verify/rollback pattern alone is Medium complexity and immediately useful.
- **Target layer:** Optimisation (closed-loop)
- **Note on recency:** Published Dec 2025 — slightly outside strict 60-day window but included due to high architectural relevance and no prior coverage of this specific pattern.

## Finding 4

### Farnebäck Dense Optical Flow Consistency as Temporal Smoothness Metric

- **Source:** PhysVideoGenerator (arXiv:2601.03665, Jan 7, 2026) + PhyCoBench (arXiv:2502.05503)
- **Relevance:** Provides a computationally cheap temporal smoothness signal for our L1 layer using standard OpenCV.
- **Key transferable method:** Compute dense optical flow between consecutive frames using Farnebäck's algorithm. Then compute flow consistency: `flow_consistency = mean(|flow(t→t+1) - flow(t-1→t)|)`. High consistency = smooth motion. Low consistency = jitter/flicker. Unlike pixel-level comparisons, this captures *motion coherence* — whether things that are moving continue to move in the same direction at the same speed.
- **How we'd use it:** For our 320×1 signal, reshape to Nx1 or small 2D (e.g. 32×10) and compute Farnebäck flow. The flow field captures "which pixels moved where" between frames. Sudden reversals or random flow vectors indicate visual noise. We can define: `jitter_score = std(flow_magnitude) / mean(flow_magnitude)` — low ratio = smooth, high ratio = jittery.
- **Implementation complexity:** Low — Farnebäck is in OpenCV (`cv2.calcOpticalFlowFarneback`). The computation is fast for our small frame size.
- **Target layer:** L1 (temporal coherence)
