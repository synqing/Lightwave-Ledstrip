## Finding 1: AdaEvolve — Hierarchical Adaptive Optimisation with Bandit-Based Budget Allocation

**Paper:** arXiv:2602.20133 — "AdaEvolve: Adaptive LLM Driven Zeroth-Order Optimization" (Feb 23, 2026)
**Authors:** Mert Cemri, Shubham Agrawal, Akshat Gupta, Shu Liu, Audrey Cheng, Qiuyang Mang, Ashwin Naren, Lutfi Eren Erdogan, Koushik Sen, Matei Zaharia, Alex Dimakis, Ion Stoica

**One-sentence relevance:** Introduces a three-level hierarchical adaptive optimisation framework with bandit-based budget allocation across parameter groups and automatic stall detection — the budget allocation and stall-detection patterns are directly transferable to our closed-loop parameter optimisation.

**Key transferable method:**
- **Three-level hierarchy:**
  1. **Local adaptation:** Dynamically adjusts exploration intensity within a parameter group based on an accumulated improvement signal. When recent iterations yield diminishing returns, exploration widens; when improvement is found, exploitation intensifies.
  2. **Global adaptation (bandit-based scheduling):** Uses a multi-armed bandit to allocate evaluation budget across different parameter groups/populations. Groups that are currently yielding the most quality improvement receive more evaluation cycles.
  3. **Meta-guidance:** When progress stalls across all groups, the system triggers a strategy diversification step — generating novel solution tactics by analysing which previous modifications succeeded vs. failed.
- **Accumulated improvement signal:** The core adaptation metric is not the absolute quality score but the *delta of quality over recent iterations*. This signal drives all three adaptation levels.
- **No manual thresholds:** The framework self-calibrates its exploration-exploitation balance based on observed improvement dynamics, eliminating the need to hand-tune mutation rates, step sizes, or exploration schedules.
- **Empirical validation:** Consistent improvement over baselines across 185 diverse optimisation problems, demonstrating robustness of the adaptive pattern across problem domains.

**Concrete application to our system:**
- **Local level:** For each parameter group (colour parameters, timing parameters, spatial parameters, decay parameters), adjust the proposal mutation magnitude based on whether recent parameter changes in that group improved quality. If colour changes have been yielding gains, make larger colour moves; if they've plateaued, shrink the step size.
- **Global level (key novelty vs. existing findings):** Use a bandit (UCB or Thompson Sampling) to decide *which parameter group to optimise next*. If timing parameters are currently the bottleneck (biggest quality improvement opportunity), allocate more of the evaluation budget to timing proposals. This is a concrete mechanism our system currently lacks — we have no principled way to decide whether to spend the next evaluation cycle tuning colour, timing, or spatial parameters.
- **Meta level:** When the improvement signal flattens across all parameter groups, trigger a more radical exploration — switch visualisation style entirely, change the colour mapping approach, or reset to a different starting configuration. This prevents the system from getting stuck in a local optimum.
- The accumulated improvement signal is trivially computed from our existing metric logs: it's a rolling average of quality score deltas.

**Implementation complexity:** Low-Medium — the bandit-based scheduling and improvement signal tracking are straightforward. The LLM-as-mutation-operator part of the paper is NOT transferable (domain-specific to their use case), but the three-level adaptive hierarchy is a clean architectural pattern applicable with any proposal engine.

**Target layer:** Optimisation (closed-loop strategy — budget allocation and stall detection)

**Differentiation from existing findings:**
- TALBO (2026-03-09_0900.md) handles drifting objectives via temporal awareness in the surrogate model. AdaEvolve handles a different problem: *how to allocate evaluation budget across parameter groups* and *when to escalate exploration intensity*.
- BoGA (2026-03-09_0900.md) handles proposal diversity via GA. AdaEvolve handles adaptive scheduling of which parameter groups to explore.
- VisionDirector (2026-03-09_2010.md) handles per-goal verification and rollback. AdaEvolve handles per-group budget allocation and stall detection.
- These four patterns are complementary and could be composed: BoGA for diverse proposals → AdaEvolve's bandit for group-level budget allocation → VisionDirector for per-goal verification → TALBO for temporal adaptation.

## Finding 2: Latent-State Bandits — Model-Free Non-Stationary Parameter Selection via Lagged Features

**Paper:** arXiv:2602.05139 — "Adaptive Exploration for Latent-State Bandits" (Feb 4, 2026, revised Feb 17, 2026)
**Authors:** Jikai Jin, Kenneth Hung, Sanath Kumar Krishnamurthy, Baoyi Shi, Congshan Zhang

**One-sentence relevance:** Provides a simpler, model-free alternative to TALBO for handling non-stationary optimal parameters — uses lagged contextual features and coordinated probing to track latent states without requiring an explicit temporal model, reducing implementation complexity from Medium-High to Low-Medium.

**Key transferable method:**
- **Lagged contextual features:** Instead of building an explicit model of how the optimal parameters evolve over time (as TALBO does with a GP-prior VAE), this approach simply uses the last N observations (audio feature vectors + recent quality scores) as context for the next parameter decision. The history implicitly encodes the current latent state.
- **Coordinated probing:** Strategically selects "informative" parameter configurations that help disambiguate which latent state the system is in. Rather than random exploration, probing actions are chosen to maximise information about the hidden state.
- **State-model-free:** No explicit model of hidden states or state transitions is needed. The algorithm learns to associate lagged feature patterns with optimal actions directly, sidestepping the GP-prior VAE requirement.
- **Robust adaptation to non-stationary rewards:** The algorithms adapt to shifting reward patterns (i.e., what constitutes "good" LED parameters changes as the music evolves) without requiring retraining or model updates.

**Concrete application to our system:**
- Our audio pipeline produces real-time features: 8-band spectral energy, tempo, beat phase, chord type, style classification, saliency scores. These features change as the music evolves.
- The optimal LED parameters (brightness, decay, colour mapping, transition speed) depend on the current musical context — but also on latent factors not fully captured by our features (e.g., "song section progression", "emotional arc", "audience energy level").
- **Lagged feature approach:** Rather than trying to model these latent factors, use the last N audio feature vectors (e.g., last 2-4 seconds) as context for parameter selection. The temporal pattern in the lagged features implicitly encodes the latent state (e.g., a crescendo shows up as a rising energy trend in the lagged features).
- **Coordinated probing:** Occasionally insert "diagnostic" parameter configurations that help determine whether the current context is a "build-up", "drop", "ambient section", etc. These probes are cheap (one frame at 120 FPS is ~8ms) and provide information that improves subsequent parameter selection.
- **Practical advantage over TALBO:** TALBO requires implementing a GP-prior VAE and integrating it with BO — a Medium-High complexity task. The lagged-feature bandit approach requires only: (a) a sliding window of recent audio features, (b) a contextual bandit algorithm (e.g., LinUCB) using the concatenated window as context. This is Low-Medium complexity.

**Implementation complexity:** Low-Medium — requires a contextual bandit algorithm (LinUCB or similar, well-documented with many open-source implementations) and a feature window buffer. No model training required; the bandit learns online from parameter→quality observations.

**Target layer:** Optimisation (closed-loop strategy — non-stationary parameter selection)

**Differentiation from TALBO (2026-03-09_0900.md):**
- TALBO: temporal GP-prior VAE, learns how the objective landscape evolves, Medium-High complexity, higher modelling power.
- Latent-state bandits: lagged features + contextual bandits, no explicit temporal model, Low-Medium complexity, faster to deploy.
- These represent two points on the complexity-capability tradeoff for the same problem (non-stationary parameter selection). Start with lagged-feature bandits for rapid deployment; upgrade to TALBO if the simpler approach hits a ceiling.
