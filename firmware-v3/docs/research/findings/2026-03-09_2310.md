## Finding 1: TP-GRPO — Step-Level Incremental Rewards with Turning Point Detection for Optimisation Trajectories

**Paper:** arXiv:2602.06422 — "Alleviating Sparse Rewards by Modeling Step-Wise and Long-Term Sampling Effects in Flow-Based GRPO" (Feb 6, 2026)
**Authors:** Yunze Tong, Mushui Liu, Canyu Zhao, Wanggui He, Shiyi Zhang, Hongwei Zhang, Peng Zhang, Jinlong Liu, Ju Huang, Jiamang Wang, Hao Jiang, Pipei Huang

**One-sentence relevance:** Introduces per-step incremental rewards and automatic turning-point detection within optimisation trajectories — directly transferable to identifying which specific parameter changes in our closed-loop have the most impact on quality.

**Key transferable method:**
- **Step-level incremental rewards:** Rather than assigning a single outcome-based reward to an entire parameter optimisation trajectory (e.g., "the final quality score was 0.82"), compute incremental quality deltas at each step. Each parameter change gets its own reward signal: Δ_quality(step_i) = Q(params after step_i) − Q(params before step_i). This decomposes the overall quality improvement into per-step contributions.
- **Turning point detection:** Identify steps where the incremental reward sign flips (positive→negative or negative→positive). Detection is fully automatic via sign changes in incremental rewards — no hyperparameters to tune. Turning points mark parameter changes that had *delayed* impact: a change that initially looked neutral or negative but led to subsequent improvements (or vice versa).
- **Long-term reward aggregation at turning points:** When a turning point is detected, the subsequent quality trajectory is aggregated and attributed back to the turning-point step. This captures causal attribution: "changing the decay parameter at step 7 initially looked neutral, but it enabled large improvements in steps 8–12."

**Concrete application to our system:**
- **Per-step attribution in the PROPOSE→APPLY loop:** Currently, our optimisation loop evaluates the *outcome* (quality after applying a batch of parameter changes). TP-GRPO's pattern says: evaluate after *each individual* parameter change to isolate which change drove the improvement. If we propose 5 parameter modifications, apply them one at a time with quality evaluation between each. The per-step deltas tell us exactly which modifications helped and which hurt.
- **Turning point detection for parameter interaction discovery:** When a parameter change causes a sign flip in the quality trajectory, it indicates parameter interaction — this change affected how subsequent changes behaved. This is a signal that these parameters are coupled and should be co-optimised, not tuned independently.
- **Computational trade-off:** More evaluations per optimisation cycle (one per parameter change instead of one per batch). Mitigated by: (a) only do step-level evaluation during exploratory phases, batch evaluation during exploitation, and (b) use cheap L0–L2 metrics for per-step evaluation, reserving expensive L3–L4 for outcome evaluation.
- **Turning point history as a meta-learning signal:** Over many optimisation cycles, the turning-point log reveals which parameter groups consistently create turning points — these are the high-leverage parameters worth optimising. This complements AdaEvolve's bandit-based budget allocation (2026-03-09_2241.md) by providing empirical evidence for which groups deserve more budget.

**Implementation complexity:** Medium — requires per-step evaluation (additional overhead per optimisation cycle) and a turning-point detection module (straightforward: sign-change detection on the incremental reward series). The causal attribution is the harder part but can start simple (just the sign-change detection).

**Target layer:** Optimisation (closed-loop — per-step attribution, parameter interaction discovery, causal analysis)

**Differentiation from existing findings:**
- AdaEvolve (2026-03-09_2241.md) handles *budget allocation across parameter groups* — which group to optimise next. TP-GRPO handles *attribution within a group* — which specific changes mattered.
- VisionDirector handles *per-goal verification and rollback*. TP-GRPO handles *per-step reward decomposition* — a finer granularity.
- TALBO handles *temporal drift* in the objective. TP-GRPO handles *trajectory-level analysis* of a single optimisation run.
- These compose naturally: TP-GRPO for per-step attribution → AdaEvolve for group budget allocation informed by attribution data → TALBO for adapting to temporal drift in the evaluation landscape.

**Why HIGH-VALUE:** This is the first finding that addresses *causal attribution within an optimisation trajectory*. All previous findings treat evaluation as outcome-based (evaluate the result). TP-GRPO evaluates the *process* — which individual changes drove improvement. For our system, this means we can learn not just "these parameters are good" but "changing parameter X was the critical modification." This accelerates convergence and builds an interpretable optimisation history.

## Finding 2: LAGO — Local-Global Competition Mechanism for Surrogate-Assisted Optimisation

**Paper:** arXiv:2603.02970 — "LAGO: A Local-Global Optimization Framework Combining Trust Region Methods and Bayesian Optimization" (Mar 3, 2026)
**Authors:** Eliott Van Dieren, Tommaso Vanzan, Fabio Nobile

**One-sentence relevance:** Introduces an adaptive competition mechanism between local trust-region exploitation and global Bayesian exploration — directly applicable to our closed-loop for deciding when to fine-tune nearby parameters vs explore radically different configurations.

**Key transferable method:**
- **Dual proposal generation:** At each iteration, *both* a local optimiser (trust-region method operating near the current best) and a global optimiser (Bayesian acquisition function over the full parameter space) independently propose candidate parameter configurations.
- **Adaptive competition:** The two proposals compete based on predicted improvement. If the local proposal is better, exploit (make small refinements near the current best). If the global proposal is better, explore (jump to a different region of parameter space).
- **Separation of exploration/exploitation at the proposal level:** The BO acquisition function is constrained to operate *outside* the trust region. This prevents redundant proposals — the local and global optimisers search different parts of the parameter space.
- **Selective information sharing:** Local evaluations are only incorporated into the global GP surrogate when they meet a minimum-distance criterion (based on GP lengthscale). This prevents the global model from being distorted by dense local sampling in a small region.
- **Gradient-enhanced GP:** The global surrogate uses gradient information from function evaluations for better modelling with fewer samples.

**Concrete application to our system:**
- **Local optimiser:** Once we have a parameter configuration that produces decent quality (say, top-25%), spin up a local trust-region optimiser that makes small, targeted tweaks. This is fast convergence in a promising region — tuning the exact decay rate, the precise colour temperature, the exact timing offset.
- **Global optimiser:** Simultaneously, a BO acquisition function explores the full parameter space for potentially better regions we haven't tried — different colour palettes, different spatial patterns, different beat-response curves.
- **Competition per cycle:** Each optimisation cycle, both propose a candidate. Evaluate the more promising one. Over time, the system naturally shifts from exploration (when the landscape is unknown) to exploitation (when a good region is found) without needing a manual schedule.
- **Interaction with TP-GRPO:** When in local exploitation mode, use TP-GRPO's per-step attribution to identify which local tweaks matter. When in global exploration mode, use batch evaluation for efficiency.

**Implementation complexity:** Medium — requires maintaining two proposal engines (a local gradient-free trust-region method and a GP-based BO), plus the competition selection logic. The GP surrogate is the most complex component but can use existing libraries (GPyTorch/BoTorch).

**Target layer:** Optimisation (closed-loop strategy — exploration vs exploitation, proposal generation)

**Differentiation from existing findings:**
- TALBO (2026-03-09_0900.md) uses temporal awareness in the surrogate to handle drifting objectives. LAGO handles the *local vs global* exploration trade-off — a different axis entirely.
- AdaEvolve (2026-03-09_2241.md) handles budget allocation *across parameter groups*. LAGO handles exploration strategy *within the full parameter space*.
- BoGA (2026-03-09_0900.md) uses GA for diverse proposals. LAGO uses *competing* local and global optimisers — a cleaner separation of exploitation and exploration.
- LAGO and AdaEvolve complement: LAGO decides local-vs-global exploration, AdaEvolve decides which parameter group gets the budget.

## Finding 3: FlowPortrait — Composite Reward Architecture Combining MLLM Signals with Hard Metric Regularisers

**Paper:** arXiv:2603.00159 — "FlowPortrait: Reinforcement Learning for Audio-Driven Portrait Video Generation" (Feb 25, 2026)
**Authors:** Weiting Tan, Andy T. Liu, Ming Tu, Xinghua Qu, Philipp Koehn, Lu Lu

**One-sentence relevance:** Demonstrates a concrete, empirically validated architecture for combining MLLM aesthetic judgements with perceptual and temporal consistency regularisers into a stable composite reward — directly applicable to constructing our L4 evaluation layer's composite quality score.

**Key transferable method:**
- **Multi-dimensional MLLM assessment:** A multimodal LLM evaluates three specific dimensions independently: (1) lip-sync accuracy (analogous to our beat-alignment accuracy), (2) expressiveness (analogous to visual diversity/interest), (3) motion quality (analogous to spatial coherence). Each dimension gets its own score.
- **Hard metric regularisers:** The MLLM scores are *not used alone*. They are combined with perceptual similarity metrics (anchoring to a reference) and temporal consistency regularisers (penalising frame-to-frame jitter). These hard metrics act as guardrails preventing the optimiser from chasing MLLM scores at the expense of perceptual quality.
- **Stable composite reward:** The composite reward = weighted_sum(MLLM_dimensions) + λ_perceptual × perceptual_reg + λ_temporal × temporal_reg. The regularisers prevent reward hacking: the system cannot achieve a high score by producing outputs that trick the MLLM while being perceptually poor.
- **GRPO for post-training:** Group Relative Policy Optimization uses the composite reward to improve the generator. The key insight: relative ranking within a group is more stable than absolute scores, making GRPO suitable for MLLM-derived rewards (which have high variance).

**Concrete application to our system:**
- **Composite quality score construction:** Our L4 layer currently faces the question: how do we combine MLLM aesthetic judgements with our L0–L3 hard metrics? FlowPortrait's answer: use L0–L2 metrics *as regularisers* on the MLLM score, not as independent signals to be averaged.
- Specifically: Composite = α × MLLM_aesthetic_score + λ₁ × L1_temporal_penalty + λ₂ × L2_beat_alignment_penalty. The MLLM drives the primary quality signal; the hard metrics prevent degeneracy.
- **Relative ranking via GRPO pattern:** When evaluating multiple parameter configurations in a batch, use group-relative ranking (which configuration is best *within the batch*) rather than absolute scores. This naturally handles MLLM score variance — even if absolute scores fluctuate across calls, the *ranking* within a batch is more stable.
- **Combines with CAMEL (2026-03-09_2213.md):** CAMEL's confidence gating decides *when* to invoke the MLLM. FlowPortrait's composite architecture decides *how to use* the MLLM signal once obtained. CAMEL gates the call; FlowPortrait structures the result.

**Implementation complexity:** Low — the composite reward is a weighted sum with regularisers. The GRPO ranking pattern is a comparison operation. Main tuning effort: calibrating the regulariser weights (λ₁, λ₂) to balance aesthetic freedom vs metric compliance.

**Target layer:** L4 (aesthetic evaluation) + Optimisation (reward construction for closed-loop)

**Differentiation from existing findings:**
- CAMEL (2026-03-09_2213.md) addresses *when to trust* the MLLM judge. FlowPortrait addresses *how to structure* the reward signal from the judge.
- VideoJudge (2026-03-09_0900.md) addresses *how to make a small judge accurate*. FlowPortrait addresses *how to combine any judge's output with hard metrics* for stability.
- These three form a complete L4 pipeline: VideoJudge for bootstrapping a small judge → CAMEL for confidence-gated evaluation → FlowPortrait for composite reward construction.

## Finding 4: EduVQA S2D-MoE — Structured Multi-Dimensional Quality Assessment with Inter-Dimension Dependencies

**Paper:** arXiv:2603.03066 — "EduVQA: Benchmarking AI-Generated Video Quality Assessment for Education" (Mar 3, 2026)
**Authors:** Baoliang Chen, Xinlong Bu, Lingyu Zhu, Hanwei Zhu, Xiangjie Sui

**One-sentence relevance:** Introduces a Structured 2D Mixture-of-Experts module that models dependencies between quality sub-dimensions and overall quality — transferable to understanding how our L0–L4 metric layers interact and influence overall perceived quality.

**Key transferable method:**
- **Multi-dimensional quality decomposition:** Rather than a single quality score, the system produces separate scores for spatial fidelity, temporal fidelity, and prompt alignment. Each sub-dimension has its own expert pathway.
- **Structured 2D MoE:** A 2D gating matrix dynamically routes quality signals between sub-dimension experts and a shared overall-quality expert. The 2D structure captures: (a) which sub-dimensions are most relevant for a given input, and (b) how sub-dimensions interact to produce the overall quality.
- **Shared experts:** Some experts are shared across sub-dimensions, capturing common quality factors. Dimension-specific experts capture what's unique to each sub-dimension.
- **Key insight:** Overall quality is NOT a simple weighted average of sub-dimensions. The gating matrix learns non-linear dependencies (e.g., temporal quality matters more when spatial quality is high; alignment matters more when perceptual quality is low).

**Concrete application to our system:**
- **Metric pipeline dependency modelling:** Our L0–L4 layers currently produce independent scores. The S2D-MoE pattern suggests we should model their interactions: does L2 (audio-visual alignment) matter more when L1 (temporal coherence) is already high? Does L4 (aesthetic quality) weight differently depending on L0 (frame quality)?
- **Practical implementation:** We likely won't train a full MoE model (insufficient data). But the *insight* — that metric interactions are non-linear and context-dependent — suggests our composite quality function should be more than a weighted sum. A simple version: adaptive weighting where λ_i depends on the values of other metrics. When temporal coherence (L1) is poor, weight it heavily (fix the obvious problem first). When L1 is good, shift weight to L2 and L4 (refinement mode).
- **Diagnostic value:** If we ever build an end-to-end quality predictor for rapid evaluation (avoiding per-layer metric computation), the S2D-MoE architecture would be the appropriate model structure because it preserves interpretability per sub-dimension.

**Implementation complexity:** Low (for the adaptive weighting insight) to High (for a full S2D-MoE model). The immediately actionable item is the adaptive-weighting pattern, not the full architecture.

**Target layer:** Architecture (metric pipeline design — how L0–L4 combine into overall quality)
