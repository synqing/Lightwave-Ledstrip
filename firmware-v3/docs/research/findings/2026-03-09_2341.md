## Finding 1: CyclicJudge — Variance Decomposition and Optimal Judge Allocation for MLLM-as-Judge Evaluation

**Paper:** arXiv:2603.01865 — "CyclicJudge: Mitigating Judge Bias Efficiently in LLM-based Evaluation" (March 2, 2026)
**Authors:** Ziyi Zhu, Olivier Tieleman, Alexey Bukhtiyarov, Jinghong Chen

**One-sentence relevance:** Provides a formal variance decomposition that partitions evaluation score variance into four isolable components and proves that round-robin judge assignment is the optimal allocation strategy — directly applicable to reducing evaluation variance in our L4 aesthetic MLLM-as-judge layer.

**Key transferable method:**

1. **Four-component variance decomposition:** Total benchmark score variance is decomposed into:
   - **Scenario variance:** Inherent difficulty differences between evaluation inputs (for us: different audio segments naturally produce different quality ceilings)
   - **Generation variance:** Stochastic variation in what's being evaluated (for us: the LED frame sequence itself, given the same audio)
   - **Judge variance:** Systematic bias of the evaluation method (for us: prompt template bias, temperature variation, or model choice in MLLM calls)
   - **Residual variance:** Irreducible noise

2. **CyclicJudge allocation:** Rather than using the same MLLM prompt/configuration for all evaluations (which conflates judge bias with quality signal), assign judges in a round-robin pattern. Each "judge" (prompt template, MLLM temperature setting, or evaluation rubric variant) evaluates different scenarios in rotation. This provably eliminates judge bias while maintaining the same evaluation cost as single-judge evaluation.

3. **Key empirical finding:** Judge bias magnitude is "similar in magnitude to the model differences that benchmarks are designed to detect." Translation for our system: MLLM prompt template choice may be introducing as much variation into L4 aesthetic scores as the actual quality differences between LED visualisation parameter configurations. If true, this means our L4 signal is substantially noisier than we assume.

**Concrete application to our system:**

- **L4 aesthetic evaluation layer:** We currently use MLLM-as-judge for aesthetic evaluation of LED frame sequences. If we call the MLLM with a single prompt template, we inherit that template's systematic bias. CyclicJudge says: maintain 3-4 prompt template variants (e.g., "rate the visual appeal", "evaluate the artistic quality", "assess the aesthetic coherence", "judge the visual-audio harmony") and cycle them round-robin across evaluation calls. The per-prompt bias cancels out over multiple evaluations.

- **Variance budget analysis:** Before deploying expensive L4 evaluations in the optimisation loop, run the variance decomposition on a held-out set. If judge variance dominates (>30% of total), the L4 signal is unreliable for fine-grained parameter comparison and we should either: (a) apply CyclicJudge to reduce it, (b) increase evaluation sample size, or (c) fall back to cheaper L0-L2 metrics for that optimisation phase.

- **Cost-neutral improvement:** The critical insight is that CyclicJudge doesn't require MORE evaluations — it requires DIFFERENT evaluations. Same number of MLLM calls, but distributed across prompt variants. This is free improvement in signal quality.

- **Composability with M-JudgeBench (Finding 2):** CyclicJudge handles judge allocation; M-JudgeBench handles judge capability assessment. Together: first assess which prompt templates are capable judges (M-JudgeBench-style capability evaluation), then cycle the qualified ones (CyclicJudge-style allocation).

**Implementation complexity:** Low — requires maintaining multiple prompt template variants (trivial) and a round-robin scheduler (trivial). The variance decomposition analysis is a one-time diagnostic that requires running the existing evaluation pipeline on a test set with multiple judge configurations.

**Target layer:** L4 (aesthetic evaluation — judge variance reduction and allocation optimisation)

**Why HIGH-VALUE:** This is the first finding that provides a formal framework for understanding WHERE evaluation noise comes from in our MLLM-as-judge pipeline. All previous aesthetic evaluation findings focused on WHAT to evaluate (rubrics, criteria, scoring schemes). CyclicJudge addresses HOW to allocate evaluation resources to maximise signal-to-noise ratio. For a closed-loop optimisation system, evaluation noise directly limits how fine-grained the optimiser can be — reducing judge variance means the optimiser can detect smaller quality improvements, enabling tighter convergence.

## Finding 2: M-JudgeBench / Judge-MCTS — Capability-Oriented Assessment and Calibration of MLLM Judges

**Paper:** arXiv:2603.00546 — "Advancing Multimodal Judge Models through a Capability-Oriented Benchmark and MCTS-Driven Data Generation" (February 28, 2026)
**Authors:** Zeyu Chen, Huanjin Yao, Ziwang Zhao, Min Yang

**One-sentence relevance:** Introduces a structured 10-subtask framework for assessing MLLM judge reliability and an MCTS-driven method for generating calibration pairs — directly transferable to validating and calibrating our L4 aesthetic MLLM judge before deploying it in the optimisation loop.

**Key transferable method:**

1. **Three-task decomposition of judge capability:**
   - **Pairwise CoT comparison** (4 subtasks): Can the judge correctly rank two responses differing in quality? Tested with short/long reasoning from same/different sources.
   - **Length bias avoidance** (3 subtasks): Does the judge prefer verbose responses regardless of correctness? Critical for our use case where a "more elaborate" textual evaluation doesn't mean a better quality score.
   - **Process error detection** (3 subtasks): Can the judge identify visual perception mistakes, logical reasoning fallacies, and incidental errors in evaluation chains?

2. **Judge-MCTS training data generation:** Uses Monte Carlo Tree Search to systematically generate paired evaluation examples with known quality orderings: short-correct, short-error, long-correct, long-error. These are paired combinatorially to create diverse calibration signals.

3. **Two-stage judge improvement:** SFT on pairwise samples + MCTS-augmented samples, followed by RL (DAPO algorithm) on mixed preference data. This produces judges that are more calibrated and less biased.

**Concrete application to our system:**

- **L4 judge validation protocol:** Before trusting an MLLM as an aesthetic judge in our optimisation loop, run a M-JudgeBench-style validation: generate LED frame sequence pairs where quality ordering is KNOWN (e.g., same audio, one with good parameters, one with deliberately degraded parameters). Test whether the MLLM judge correctly ranks them. Also test with pairs that differ only in description verbosity (does the judge prefer longer MLLM-generated evaluation text?).

- **Calibration pair generation:** Adapt Judge-MCTS to generate LED visualisation evaluation pairs. Instead of reasoning traces, generate frame sequences with systematically varied quality levels. Use these to calibrate the judge's scoring threshold — ensuring a score of 7.0 means the same thing across different audio segments and parameter configurations.

- **Process error detection for our domain:** Our L4 judge produces CoT-style evaluations ("The colours transition smoothly... The beat alignment is accurate..."). Process error detection checks whether these intermediate reasoning steps are actually correct — does the MLLM claim "smooth transition" when the frames show flickering? This catches hallucinated evaluations.

**Implementation complexity:** Medium — requires generating calibration pairs (needs the existing evaluation pipeline to produce frame sequences with known quality levels) and running the validation protocol (additional MLLM calls, but one-time setup cost).

**Target layer:** L4 (aesthetic evaluation — judge calibration and validation)

**Differentiation from existing findings:**
- CyclicJudge (Finding 1) handles WHERE to allocate evaluations. M-JudgeBench handles WHETHER the judge is trustworthy in the first place.
- Calibrating MLLM-as-a-judge via Multimodal Bayesian Prompt Ensembles (arXiv:2509.08777, already known) handles prompt weighting. M-JudgeBench handles capability assessment across structured subtasks.
- Composition: M-JudgeBench for judge validation → CyclicJudge for judge allocation → Bayesian Prompt Ensembles for prompt weighting. This is a complete L4 judge quality pipeline.
