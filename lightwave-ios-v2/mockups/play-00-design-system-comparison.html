iOS Companion
Display: 6.1" Super Retina OLED, 2556 x 1179 px

Viewing distance: 25-40cm (hand-held)
Input: Touch (thumb-reachable zones)
Environment: Mixed lighting conditions

Accent: #FFB84D (warm gold, softer)

Key Differences That Drive Token Divergence

LCD vs OLED: Tab5's IPS LCD has a backlight bleed floor — #08080A appears as near-black but is still backlit. OLED pixels turn fully off, so #000000 and #08080A are indistinguishable on iPhone. The iOS choice of #0F1219 (with a blue shift) adds visual warmth on OLED without sacrificing depth.

Viewing angle: Tab5 is stationary at arm's length; iOS is hand-held at variable distance. iOS needs larger touch targets (44pt minimum per Apple HIG) and slightly larger body text to accommodate motion while reading.

PPI density: At 460 PPI (iPhone) vs 292 PPI (Tab5), the same font size renders with more physical detail on iOS. Tab5's larger display sizes (44px hero) compensate for lower PPI and greater viewing distance.

Contrast Analysis Summary

Tab5 failures: FG_DIMMED (#3A3F4A) fails on both BG_DEEP (2.7:1) and SURFACE_ELEVATED (1.9:1). This is intentional — dimmed text is designed to be barely visible on the small Tab5 screen, functioning as a visual "off" state rather than readable text. On a mobile device, this would be an accessibility violation.

iOS strengths: The iOS tertiary text lwTextTertiary (#6B7280) passes AA on the base background (4.6:1) but fails on elevated surfaces (2.7:1). This is the only failing pair in the iOS system. The higher luminance of iOS dimmed text makes it readable as actual content rather than decorative.

Brand accent: Both accents pass AAA on their respective backgrounds. Tab5's #FFC700 on #08080A achieves 10.6:1, while iOS's #FFB84D on #0F1219 achieves 8.3:1. Both are excellent.

Typography Scale Analysis

Scale ratio: Tab5 sizes are roughly 1.5-1.8x larger than iOS equivalents in absolute pixels, which is correct. At arm's length on a 292 PPI screen, Tab5's 20px label subtends a similar visual angle to iOS's 12px label at hand-held distance on a 460 PPI screen. The scaling is physically justified.

Font choice divergence: Tab5 uses JetBrains Mono Bold for the BPM hero metric; iOS uses Bebas Neue Bold. This is the one place where font assignment (not just size) differs. Tab5's choice provides monospace digit alignment for rapid BPM changes; iOS's Bebas Neue gives a bolder, more display-oriented look. Both are valid for their context.

iOS minimum of 10px: This is at the lower boundary of Apple's recommended minimum. At 460 PPI it renders crisply, but users with accessibility settings (Dynamic Type) may need it to scale. The relativeTo: parameter in DesignTokens.swift enables this correctly.

Weight divergence: Tab5 uses Bebas Neue Regular for headlines; iOS uses Bold. The bold weight compensates for the smaller physical size on iPhone, maintaining visual weight parity. This is a good adaptation.

Recommendation
Should the iOS app adopt Tab5 tokens exactly, adapt them, or keep its own?
Option A: Adopt Tab5 Exactly

Use all Tab5 tokens verbatim on iOS.

  • Tab5's #08080A background is indistinguishable from true black on OLED — loses the subtle navy warmth that differentiates the app from "just black"
  • Tab5's FG_DIMMED (#3A3F4A) fails WCAG AA on mobile — fine for a decorative "off" state on a tiny screen, unacceptable for iOS where users expect readable text
  • Tab5 has no BORDER_SUBTLE equivalent for card separation on larger screens
  • Tab5's pure #FFC700 gold can feel harsh on OLED at close viewing distance
  • Typography sizes are for 2.4" @ 292 PPI — 44px heroes would be comically large on iPhone
Option B: Unified Semantic Tokens (Recommended)

Define a shared semantic token vocabulary with platform-specific values. Same names, adapted colours.

  • Both platforms use the same token names: lw.bg, lw.surface, lw.elevated, lw.brand, lw.fg.primary, lw.fg.secondary, lw.fg.tertiary
  • Backgrounds: Tab5 uses neutral ultra-dark (#08080A), iOS uses navy-dark (#0F1219) — both are "the darkest background" on their display technology
  • Brand: converge on #FFC700 as the canonical brand gold. iOS may tint to #FFB84D for OLED warmth, but the token is the same semantic role
  • Status colours: converge on Tab5's standard set (#22C55E, #F59E0B, #EF4444) which are industry-standard and work on both displays
  • Typography: keep the 3-family stack, adapt sizes per platform. Document the "physical visual angle" relationship, not absolute pixels
  • iOS retains lwCyan and zone colours as platform extensions — Tab5 does not need them (no zone visualisation on 2.4" screen)
Option C: Keep Completely Separate

Maintain independent token sets with no shared vocabulary.

  • Risk: brand fragmentation as the product ecosystem grows (Tab5, iOS, web dashboard, landing page)
  • Already diverging: background hue family, gold tone, status green, tertiary text luminance
  • Makes cross-platform mockups and design reviews harder
  • Developer confusion when implementing the same feature on both platforms

Key Decisions

Background: Keep platform-specific backgrounds. Tab5's near-black (#08080A) is correct for IPS LCD backlight bleed. iOS's navy-dark (#0F1219) is correct for OLED depth + warmth. The semantic token lw.bg maps to different values per platform, which is the standard design token pattern.

Brand gold: Converge toward #FFC700 as the canonical brand colour. iOS's #FFB84D is 30% less saturated and warmer — this was likely chosen for OLED comfort, but the saturation loss weakens brand recognition. Consider #FFBE33 as a compromise: maintains most of the #FFC700 energy while adding a touch of orange warmth for OLED. Alternatively, keep #FFC700 for brand marks and use #FFB84D only for large filled surfaces where high saturation would cause visual fatigue.

Status colours: Align on Tab5's set. The iOS #4DFFB8 success green is unusual (cyan-shifted, very bright) and may not scan as "success" to users accustomed to standard green. #22C55E is the Tailwind CSS green-500 — widely recognised, well-tested.

Tertiary text: iOS's #6B7280 is the better choice. Tab5's #3A3F4A is too dim for any context where text should be readable. The Tab5 token should remain for truly decorative/disabled elements, but add an intermediate token for "low priority but still readable" content.

Unified Token Migration Path

Phase 1 (immediate): Adopt the token naming convention lw.* in both codebases. Tab5 maps tokens to its current values; iOS maps to its current values. Zero visual change, but the vocabulary is shared.

Phase 2 (converge): Align status colours to Tab5's set. Replace iOS #4DFFB8 with #22C55E. Replace iOS #FF4D4D with #EF4444. Align secondary text to #9CA3B0 on both platforms. Add lw.fg.disabled (#3A3F4A) as an explicit "decorative/off" token on both platforms.

Phase 3 (extend): Add lw.brand.soft (#FFBE33) for OLED-safe large surfaces. Evaluate whether Tab5 should adopt the navy-tinted surfaces or keep pure neutral — this depends on whether the brand direction is "neutral instrument" or "warm product".

Typography: Do not unify sizes — they must remain platform-specific. Unify the font family assignments and semantic role names. Both already use Bebas/Rajdhani/JetBrains Mono, so brand consistency is already achieved at the typeface level.

Visual Comparison Notes

Card depth: Tab5 cards sit closer to the background (SURFACE #111114 is only 4% lighter than BG_DEEP #08080A). iOS cards have more separation (lwCard #1E2535 is 8% lighter and bluer than lwBase #0F1219). On the 2.4" Tab5 screen, subtle separation is sufficient; on a 6.1" phone, the stronger card contrast helps users parse layout structure at a glance.

Corner radius: Tab5 uses 10px corners; iOS uses 16px. iOS's larger radius follows Apple's visual language and feels native to the platform. The unified system should specify radii as semantic tokens (lw.radius.card) with platform-specific values.

Badge shape: Tab5 uses 4px radius pills (squared); iOS uses 12px (rounded). Both are correct for their platform's visual language. This is a deliberate platform adaptation, not a divergence to fix.

Battery/status visibility: Tab5's #3A3F4A "BAT 87%" is nearly invisible — intentionally de-emphasised. iOS's #6B7280 makes the same info readable. For a mobile app where battery context matters more (phone battery, not just K1 battery), the iOS approach is correct.

Design system comparison. Two platforms, one brand. The Tab5 design system is optimised for a dense, information-rich 2.4" touchscreen viewed at arm's length in dark environments. The iOS system adapts the same visual language for hand-held OLED with larger touch targets, warmer tones, and stronger card separation. Convergence should happen at the semantic token vocabulary level, not by forcing identical colour values across fundamentally different display technologies. The recommended path is Option B: unified naming, platform-specific values, with status colours and secondary text aligned for consistency.
