Frame Budget — The Non-Negotiable Numbers

At 60 fps, the Main Thread has 16.67 ms per frame. The firmware emits frames every 8 ms (125 Hz). The iOS display refreshes every 16.67 ms. All work described in this document must fit inside these bounds.

16.67 ms budget breakdown (per display frame)

Canvas draw — LED strip (320 px)
< 3 ms
Canvas draw — Spectrum (64 bins)
< 1.5 ms
Canvas draw — Waveform (160 pts)
< 1 ms
Observable property writes (per frame)
< 1 ms
SwiftUI layout / diffing overhead
< 2.5 ms
Headroom (network I/O, haptics)
> 7 ms
Key insight The firmware at 125 Hz produces two UDP frames per display frame. Only one of them needs to reach the render call. Throttling at the ViewModel boundary is therefore mandatory, not optional.
01

Canvas vs SwiftUI View Rendering

Canvas renders to a single Metal-backed layer in one draw call per frame. SwiftUI views are a retained scene graph — each child view allocates heap objects, participates in layout, and generates an Equatable diff every frame. For data that changes every frame, a retained scene graph is prohibitively expensive.

Decision Tree

Why TimelineView wraps every Canvas

Without TimelineView, a Canvas only redraws when its observed properties change. With .animation schedule, it redraws every display-sync frame, exactly matching the 60 fps ProMotion cadence. The current WaterfallCanvas, DotMatrixCanvas, and DotRidgeCanvas all use this correctly.

Anti-pattern — 320 SwiftUI views

// 320 Color views = 320 layout passes
// per frame = guaranteed jank

        Color(
            red: Double(ledData[i*3]) / 255,
            green: Double(ledData[i*3+1]) / 255,
            blue: Double(ledData[i*3+2]) / 255
        )
        .frame(width: 3, height: 24)
    }
}
Good — single Canvas draw call

    Canvas { ctx, size in
        let w = size.width / 320
        for i in 0..320 {
            let r = ledPixels[i].r
            let g = ledPixels[i].g
            let b = ledPixels[i].b

                Path(CGRect(
                    x: CGFloat(i) * w,
                    y: 0,

02

Observable Granularity — Minimising View Invalidation

With @Observable (iOS 17+), a view only re-renders when it reads a property that changes. But if all audio data lives on one large ViewModel, every property write risks invalidating views that only needed a subset of properties. The fix is to split the ViewModel into focused data objects so the observation graph is fine-grained.

How @Observable invalidation works The Observation framework tracks property access at the getter level using _$observationRegistrar. When bands is written, only views that read bands during their last render are invalidated. This is much more granular than ObservableObject/objectWillChange, but it still means that a view reading any property from a monolithic ViewModel re-renders on any write to that ViewModel. At 125 Hz, every write to AudioViewModel triggers a re-render in every view that observed it.

Current architecture (LightwaveOS)

The existing AudioViewModel (432 LOC) carries spectrum data, beat state, DSP metrics, and parameter controls on a single object. This is workable because the three Canvas-based views observe only audio.bands, audio.ringBuffer, and audio.waveform — properties that update every frame anyway. The beat-indicator views do not observe high-frequency properties, so they rarely re-render.

The pattern below shows how to take this further if profiling reveals unnecessary re-renders. It is a recommended evolution path, not a required immediate refactor.

125 Hz → 60 fps Frame Throttling

The firmware emits UDP frames at 125 Hz (one every 8 ms). The display refreshes at 60 Hz (one every 16.67 ms). Naively dispatching every frame to the Main Thread causes two problems: (a) Main Thread load doubles needlessly, and (b) beat transients can arrive mid-frame and be dropped if the throttle is too aggressive.

Architecture

The correct pattern has three stages: (1) receive on a background queue, (2) throttle using a display-sync timer, (3) write to the ViewModel only on Main Actor.

Option A — Accumulator + display-synced flush (recommended)

The nonisolated processFrame in UDPStreamReceiver receives on the UDP background queue. Instead of immediately dispatching to Main Actor, it writes into a lock-protected accumulator. A display-synced timer (CADisplayLink or TimelineView clock) reads from the accumulator once per display frame. 

// FrameAccumulator lives in UDPStreamReceiver
// Written by UDP queue, read by Main Actor.
final class FrameAccumulator: @unchecked Sendable {
    private var lock = OSAllocatedUnfairLock()
    private var pendingLEDBytes:   [UInt8]? = nil
    private var pendingAudioFrame: AudioMetricsFrame? = nil
    // Beat transients are queued — they must NOT be dropped
    private var pendingBeats:       [BeatEvent] = []

    // Called on UDP background queue — O(1), no alloc
    func storeLED(_ bytes: [UInt8]) {
        lock.withLock { pendingLEDBytes = bytes }
    }

    func storeAudio(_ frame: AudioMetricsFrame) {
        lock.withLock { pendingAudioFrame = frame }
    }

    func storeBeat(_ event: BeatEvent) {
        lock.withLock { pendingBeats.append(event) }
    }

    // Called on Main Actor, once per display frame
    func drain() -> (LEDBytes?, AudioMetricsFrame?, [BeatEvent]) {
        lock.withLock {
            let led   = pendingLEDBytes;   pendingLEDBytes = nil
            let audio = pendingAudioFrame; pendingAudioFrame = nil
            let beats = pendingBeats;      pendingBeats = []
            return (led, audio, beats)
        }
    }
}

// In AudioViewModel — flushed by TimelineView's
// .animation schedule, which fires at display cadence.
func flushAccumulator() {
    let (led, audio, beats) = accumulator.drain()
    if let frame = audio { applyAudioFrame(frame) }
    if let bytes = led   { applyLEDBytes(bytes)   }
    beats.forEach { handleBeatEvent($0) }
}

// In WaterfallCanvas (and all real-time Canvases)
TimelineView(.animation) { ctx in
    Canvas { gfx, size in
        // This callback fires at display cadence.
        // Flushing here gives us one write per render.
        viewModel.flushAccumulator()
        drawWaterfall(gfx, size)
    }
}
Beat transients must never be dropped Beats arrive at ~2 Hz (impulse events), not continuously. The accumulator queues them rather than overwriting them. A rapid double-beat (e.g., triplet feel at 180 BPM) can arrive 333 ms apart — both must fire their haptic and visual flash. Use a queue ([BeatEvent]), not a single slot, for beat data.

Option B — Minimum interval gate (simpler, slightly less precise)

// Gate writes to Main Actor using minimum interval.
// Simpler than an accumulator but may drop one extra
// frame per burst at the 125→60 boundary.
@MainActor
class AudioViewModel {
    private var lastFlushTime: TimeInterval = 0
    private let minFlushInterval: TimeInterval = 1.0 / 62  // 62 Hz guard

    func handleMetricsFrame(_ frame: AudioMetricsFrame) {
        let now = CACurrentMediaTime()
        guard now - lastFlushTime >= minFlushInterval else {
            // Still queue beats even when throttling visual data
            if frame.beatTick > 0 { handleBeatImpulse(frame) }
            return
        }
        lastFlushTime = now
        applyFullFrame(frame)
    }
}

LED Strip Preview — 320 Colour-Changing Rectangles

The LED preview must render 320 pixel-width rectangles, each with an independent RGB colour, updating every display frame. This is the highest-throughput render task in the app. The non-negotiable choice is Canvas.

Data layout — pack for the render loop

The current AppViewModel.ledData is a flat [UInt8] array with stride-3 RGB layout matching the UDP frame directly. Prefer a typed per-pixel struct to avoid multiplying indices in the hot path: 

// Pre-typed pixel buffer — zero extra allocations on decode.
// Avoids 3× index multiplication in the Canvas draw loop.
struct LEDPixel: Sendable {
    var r: UInt8
    var g: UInt8
    var b: UInt8

    // Inline color conversion — avoids bridging to Foundation Color
    @inline(__always)
    func toSwiftUIColor() -> Color {
        Color(
            red: Double(r) * (1.0 / 255),
            green: Double(g) * (1.0 / 255),
            blue: Double(b) * (1.0 / 255)
        )
    }
}

// AppViewModel — typed pixel buffer alongside the raw UInt8 array
@MainActor @Observable
class AppViewModel {
    // Existing raw bytes (keep for protocol compatibility)
    var ledData: [UInt8] = Array(repeating: 0, count: 960)

    // Typed buffer — pre-allocated, reused each frame
    var ledPixels: [LEDPixel] = Array(
        repeating: LEDPixel(r: 0, g: 0, b: 0),
        count: 320
    )

    // Called once per display frame by FrameAccumulator.drain()
    func applyLEDBytes(_ bytes: [UInt8]) {
        // Decode stride-3 bytes into typed pixel buffer
        // No allocation — overwrites existing array elements
        let count = min(bytes.count / 3, ledPixels.count)
        for i in 0..LEDPixel(
                r: bytes[i * 3],
                g: bytes[i * 3 + 1],
                b: bytes[i * 3 + 2]
            )
        }
    }
}

Memory Patterns — Pre-Allocated Buffers

The render loop executes 60 times per second. Any heap allocation inside the render path — including array creation, string interpolation, or Array() copies — causes ARC pressure that the allocator amortises across frames, introducing stutter at unpredictable intervals.

The rule: zero allocations in the Canvas draw closure The Canvas draw closure is analogous to a C++ render() function. Treat it identically: no Array(), no String(format:), no append, no conditional let x = someOptional.map { ... } that creates a new collection. All buffers must be allocated once at init time and reused in-place.

Pattern A — Fixed-size pre-allocated arrays on the ViewModel

@MainActor @Observable
final class AudioStreamData {
    // MARK: - Pre-allocated buffers
    // All buffers are allocated ONCE at init.
    // The Canvas draw closure reads these directly — no copy.

    /// 8 octave band values (sub → air), range 0.0 – 1.0
    var bands:     [Float] = Array(repeating: 0, count: 8)

    /// 160 waveform samples, normalised -1.0 ... 1.0
    var waveform:  [Float] = Array(repeating: 0, count: 160)

    /// 60 × 8 ring buffer for waterfall history
    var ringBuffer: [[Float]] = Array(
        repeating: Array(repeating: Float(0), count: 8),
        count: 60
    )
    var writeRow: Int = 0

    // MARK: - Frame application (no alloc)

    func applyFrame(_ frame: AudioMetricsFrame) {
        // Overwrite in-place — no Array() copy
        bands = frame.bands          // [Float] copy: OK (value type, fixed-size)
        waveform = frame.waveform    // [Float] copy: OK

        // Ring buffer: overwrite the slot, advance cursor
        ringBuffer[writeRow] = frame.bands
        writeRow = (writeRow + 1) % 60
    }
}

// WRONG — allocates a new [Float] on every frame:
// bands = frame.bands.prefix(8).map { $0 }   ← map() creates new array
// bands = Array(frame.bands.prefix(8))        ← Array() wrapping creates copy

// RIGHT — direct assignment of value types:
// bands = frame.bands                          ← COW, no alloc if count unchanged

Pattern B — Unsafe mutable buffer pointer for hottest paths

For the waveform Canvas (160 points drawn as a path), the path itself allocates internally. Pre-build the path into a re-used mutable variable rather than creating a new Path on every frame: 

// Re-use a single Path across frames by building it from scratch
// but using withUnsafeBytes to avoid extra Array copies during decode.
struct WaveformCanvas: View {
    let samples: [Float]  // passed by reference from pre-allocated buffer

    var body: some View {
        Canvas { ctx, size in
            guard samples.count > 1 else { return }

            // Path() is a value type — the Path initialiser closure
            // captures no heap — this is acceptable.
            let step = size.width / CGFloat(samples.count - 1)
            let midY = size.height * 0.5

            var path = Path()
            path.reserveCapacity(samples.count)  // pre-size internal storage

            for (i, s) in samples.enumerated() {
                let pt = CGPoint(
                    x: CGFloat(i) * step,
                    y: midY - CGFloat(s) * midY
                )
                if i == 0 { path.move(to: pt) } else { path.addLine(to: pt) }
            }

            ctx.stroke(path, with: .color(.lwCyan), lineWidth: 1.5)
        }
    }
}

Pattern C — Ring buffer write without inner array alloc

// Instead of [[Float]] (array of arrays), use a flat [Float]
// for the ring buffer. Eliminates inner [Float] allocation on each push.
@MainActor @Observable
final class AudioStreamData {
    static let historyCount = 60
    static let bandCount    = 8

    // Flat layout: ringFlat[row * bandCount + band]
    var ringFlat: [Float] = Array(
        repeating: 0,
        count: historyCount * bandCount
    )
    var writeRow: Int = 0

    func pushBands(_ bands: [Float]) {
        let base = writeRow * Self.bandCount
        for i in 0..min(bands.count, Self.bandCount) {
            ringFlat[base + i] = bands[i]
        }
        writeRow = (writeRow + 1) % Self.historyCount
    }

    // Canvas access — no array slice, direct index arithmetic
    @inline(__always)
    func band(historyOffset: Int, bandIndex: Int) -> Float {
        let row = (writeRow - 1 - historyOffset + Self.historyCount)
            % Self.historyCount
        return ringFlat[row * Self.bandCount + bandIndex]
    }
}

Debounce Patterns — 150 ms Without Blocking UI

Parameter sliders must debounce network updates at 150 ms minimum to prevent flooding the K1 firmware (which rate-limits REST at ~20 req/s). The constraint is: the display must not debounce — the slider should move in real time, and only the network transmission should be deferred.

Canonical pattern (used in AudioViewModel)

// Cancel-and-restart debounce using structured concurrency.
// The UI property (gain) updates immediately.
// The network write is deferred by 150 ms.
@MainActor @Observable
class AudioViewModel {
    var gain: Double = 1.0 {
        didSet { queueGainUpdate() }
    }

    private var gainDebounceTask: Task<Void, Never>?

    private func queueGainUpdate() {
        // Cancel any pending write — the slider is still moving
        gainDebounceTask?.cancel()

        gainDebounceTask = Task { @MainActor [weak self] in
            guard let self else { return }
            do {
                // Wait for slider inactivity
                try await Task.sleep(nanoseconds: 150_000_000)
                guard !Task.isCancelled else { return }

                // Now fire the REST call
                try await restClient?.setAudioParameters(gain: gain, ...)
            } catch is CancellationError {
                // Cancelled by a newer slider move — expected, discard silently
            } catch {
                print("[Audio] Gain update failed: \(error)")
            }
        }
    }
}

Debouncing multiple parameters together

When multiple related DSP parameters (band attack, band release, heavy attack, heavy release) change together in a drag gesture, debounce them as a group rather than individually. This reduces REST calls from 4 to 1 per gesture: 

@MainActor @Observable
class AudioViewModel {
    // All four parameters trigger the SAME debounce task
    var bandAttack: Double = 0.15  { didSet { queueTuningUpdate() } }
    var bandRelease: Double = 0.03 { didSet { queueTuningUpdate() } }
    var heavyAttack: Double = 0.08 { didSet { queueTuningUpdate() } }
    var heavyRelease: Double = 0.02{ didSet { queueTuningUpdate() } }

    private var audioTuningDebounceTask: Task<Void, Never>?

    private func queueTuningUpdate() {
        audioTuningDebounceTask?.cancel()
        audioTuningDebounceTask = Task { @MainActor [weak self] in
            guard let self else { return }
            do {
                try await Task.sleep(nanoseconds: 150_000_000)
                guard !Task.isCancelled else { return }
                try await restClient?.patchAudioTuning([
                    "pipeline": [
                        "bandAttack": bandAttack,
                        "bandRelease": bandRelease,
                        "heavyBandAttack": heavyAttack,
                        "heavyBandRelease": heavyRelease
                    ]
                ])
            } catch is CancellationError { }
            catch { print("[Audio] Tuning update failed: \(error)") }
        }
    }
}

Background / Foreground Lifecycle

Real-time rendering must stop when the app enters the background. Failure to stop wastes CPU on a non-visible Canvas, keeps the UDP socket open (triggering watchdog-adjacent battery drain), and causes a crash on some iOS versions when the Metal render target is released. This section covers the correct clean-stop and clean-resume pattern.

What must stop on background

  • UDP listener (UDPStreamReceiver.stop())
  • Telemetry polling tasks (AudioViewModel.stopTelemetryPolling())
  • TimelineView animation — automatic (OS pauses the app, Canvas stops receiving display-sync callbacks)
  • Beat reset tasks — cancel and clear visual state so there is no frozen "beat active" indicator on resume
  • Debounce tasks — cancel to avoid firing a REST call after the session state is stale

What must resume on foreground

  • UDP listener restart (only if connected)
  • Telemetry polling restart (only if Audio tab is visible)
  • Beat state cleared and re-synced from next incoming frame

Implementation via ScenePhase

// In the root ContentView or App struct.
// ScenePhase is the correct SwiftUI lifecycle hook.
struct ContentView: View {
    @Environment(\.scenePhase) private var scenePhase
    @Environment(AppViewModel.self) private var app

    var body: some View {
        // ... main tab view ...
        .onChange(of: scenePhase) { _, newPhase in
            switch newPhase {
            case .active:
                app.handleForeground()
            case .background:
                app.handleBackground()
            case .inactive:
                // App switcher, incoming call — pause non-essential work
                app.handleInactive()
            @unknown default:
                break
            }
        }
    }
}

@MainActor
extension AppViewModel {

    func handleBackground() {
        // 1. Stop UDP stream — conserves CPU and battery
        udpReceiver.stop()
        isLEDStreamActive = false

        // 2. Stop audio telemetry polling
        audio.stopTelemetryPolling()

        // 3. Clear transient visual state
        audio.reset()  // cancels beat reset tasks, clears isBeating

        // 4. Cancel all pending debounce tasks
        // (prevent stale REST calls when session resumes)
        audio.cancelPendingDebounces()

        log("App entered background — streams stopped")
    }

    func handleForeground() {
        guard connectionState == .connected else { return }

        // 1. Restart UDP stream
        udpReceiver.start()

        // 2. Restart telemetry only if Audio tab is active
        if currentMode == .audio {
            audio.startTelemetryPolling()
        }

        // 3. Refresh device state — may have changed while backgrounded
        Task { try await refreshDeviceState() }

        log("App returned to foreground — streams resumed")
    }

    func handleInactive() {
        // Inactive = screen lock, notification centre, app switcher.
        // Do NOT stop UDP here — the app may return active within milliseconds.
        // Only background warrants a full stop.
    }
}

Tab visibility optimisation

Telemetry polling costs ~9.5 HTTP req/s when running. If the user is on the Play tab, polling the Audio endpoint is wasteful. Stop polling on tab change, restart on Audio tab entry: 

// In AudioTab or the TabView onAppear / onDisappear
struct AudioTab: View {
    @Environment(AppViewModel.self) private var app

    var body: some View {
        ScrollView {
            // ... audio cards ...
        }
        .onAppear {
            if app.connectionState == .connected {
                app.audio.startTelemetryPolling()
            }
        }
        .onDisappear {
            // UDP stream keeps running (it feeds LED preview on Play tab)
            // but audio-specific REST polling stops
            app.audio.stopTelemetryPolling()
        }
    }
}
UDP vs REST lifecycle — they differ The UDP stream feeds both the LED preview (Play tab) and the audio visualisations (Audio tab). It should run whenever the app is connected and in the foreground, regardless of which tab is active. REST polling is Audio-tab-specific and should only run when the Audio tab is visible.

Quick Reference — Pattern Checklist

Canvas

  • Always wrap in TimelineView(.animation)
  • Zero allocations inside draw closure
  • Use Path.reserveCapacity for waveform
  • Add +0.5 width to prevent sub-pixel gaps
  • LED strip: 320 fill calls, not 320 Color views

@Observable

  • Split ViewModel by update frequency
  • Canvas views read only high-frequency properties
  • BPM/beat views read only beat properties
  • Param cards read only param properties
  • Use @Environment, not @ObservedObject

Throttling

  • Accumulator pattern for 125 Hz → 60 fps
  • Beat events queued, never overwritten
  • Drain in TimelineView callback (display-synced)
  • Use CACurrentMediaTime() for gate checks

Debounce

  • 150 ms minimum (K1 REST rate limit)
  • Cancel-restart pattern with Task.sleep
  • UI property updates immediately
  • Network write deferred
  • Group related params under one debounce task

Memory

  • All buffers allocated at init time
  • No Array(), .map {}, .prefix() in render
  • Flat ring buffer (1D) for 2D history
  • COW-safe: assign [Float] directly, not via map

Lifecycle

  • .background → stop UDP + REST + cancel tasks
  • .active → restart UDP if connected
  • REST polls restart only on Audio tab appear
  • REST polls stop on Audio tab disappear
  • Use scenePhase, not UIApplication notifications
