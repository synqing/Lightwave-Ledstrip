Design Rationale

Why Carousel Pages

At 128x128 pixels, every pixel is precious. Previous designs attempted to show connection status, zone states, effect info, and mixer parameters simultaneously, requiring text as small as 6px -- below the legibility threshold for a 0.85" panel viewed at arm's length. The carousel pattern trades density for clarity: each page shows one concern with text large enough to read at a glance.

Page Navigation

  • AtomS3 has a single built-in button (the screen itself is a button).
  • Short press cycles to the next page (1 → 2 → 3 → 4 → 1).
  • The 4-dot indicator at the bottom provides position awareness without consuming significant vertical space (8px total height including padding).
  • Auto-return to Page 1 (CONNECTION) after 30 seconds of inactivity -- connection status is the most operationally important page during live performance.

Typography Sizing

Element Size (real px) Purpose
Primary values 13-14px Effect name, mixer params -- must be readable at 50cm
IP address, zone names 10-11px Key identifiers -- readable at 40cm
Secondary info 8px Brightness, RSSI, speed -- glanceable context
Labels, tags 6-7px Category headers -- orientation, not primary reading
Page dots 4px diameter Position indicator -- pattern recognition, not text

Colour Strategy

  • Black background maximises contrast ratio on LCD (effectively infinite for OLED).
  • Each page has a distinct accent colour to build muscle memory: cyan (connection), zone-specific RGB (zones), amber (effect), violet (mixer).
  • Zone colour bars use the same saturated hues across all UI mockups for cross-design consistency.
  • ON/OFF states use green/dark-red for immediate status recognition without reading text.

Information Architecture

  • Page 1 (Connection): The most critical operational page. WiFi icon provides instant status recognition. IP address is the primary data (needed for manual WebSocket debugging). Signal bars and WS state are secondary confirmation.
  • Page 2 (Zones): Three rows, one per zone. Colour bars on the left provide zone identity. Brightness percentage is the most-adjusted parameter. ON/OFF state on the right edge for quick scanning.
  • Page 3 (Effect): Centres the effect name as the dominant element. Effect names can be long (up to ~16 characters), so the 13px size with word-wrap accommodates names like "Spectral Bloom" or "Chromatic Pulse".
  • Page 4 (Mixer): EdgeMixer mode name dominates. Spread and strength are the two primary tunable parameters, shown as large numerals. Spatial/temporal tags show which processing stage is active.

Implementation Notes for LVGL

  • Use lv_tabview or manual page index with lv_obj_set_hidden() toggling.
  • Page dots: 4x lv_obj circles in an lv_obj row container, update bg_color on page change.
  • Button ISR increments page index modulo 4, posts event to LVGL task.
  • Auto-return timer: lv_timer_create() with 30s period, reset on button press.
  • Font recommendation: lv_font_montserrat_14 for primary, lv_font_montserrat_10 for secondary. Compile both into the binary (total ~12KB flash).
  • Zone colour bars: lv_obj with bg_color set from zone config, border_width: 0, radius: 2.

Accessibility Considerations

  • All text-on-black exceeds WCAG AAA contrast (7:1+). White on black = 21:1. Dim grey (#888) on black = 5.3:1 (AA compliant).
  • Accent colours chosen for distinguishability under common colour vision deficiencies (protanopia, deuteranopia): cyan/amber/violet remain distinct.
  • Signal bars use height progression (not colour alone) to convey strength.
  • ON/OFF uses both colour and text label -- never colour alone.
