Design Rationale — Context-Aware Pages

Core Philosophy

The operator should never need to think about page navigation. Traditional multi-page UIs impose a cognitive tax: the operator must remember which page they are on, how to get to the one they need, and navigate there before adjusting anything. Context-Aware Pages eliminate this entirely. The display shows what the operator needs, precisely when they need it, driven by their physical actions on the hardware.

Auto-Switching Logic

Four context states govern the display, with deterministic transition rules:

IDLE → ZONE → IDLE  |  IDLE → ENCODER → IDLE
  • IDLE (>5s no input): Connection page. Shows WiFi status, WebSocket state, IP address, and device ID. This is the information most useful when the operator is not actively mixing — confirming connectivity before a performance, or diagnosing a dropped link.
  • ZONE (scroll wheel): Auto-switches when any of the 4 scroll wheels are turned (Zone 1/2/3 + Master). Shows all 3 zone brightness bars plus master in a single view. The operator gets immediate spatial awareness of the full brightness landscape.
  • ENCODER (8encoder rotation): Shows the specific parameter page relevant to the encoder being turned — effect selection, speed, or EdgeMixer depth. Only the active parameter is featured, preventing information overload.
  • CONFIRM (button press): 1.5s flash showing the action result. High-contrast text on a coloured background — unmissable confirmation that the command was received. Returns to previous context after the flash.

Live Overlay System

Orthogonal to the page system, the live overlay fires for 2 seconds whenever any value changes. It appears as a translucent bar across the bottom third of the display, showing the parameter name and its new value with a proportional fill bar. This means the operator always sees immediate feedback even if the page context does not change. The overlay is non-modal — it layers on top of whatever page is active and disappears automatically.

Why Not Manual Page Cycling?

  • Fewer controls needed. No page-up/page-down button required. The AtomS3 button can be used for more valuable functions (preset save, zone toggle).
  • Zero navigation latency. The display is always in the right context. No “scroll to page 3” delay before adjusting a parameter.
  • Reduces mode errors. The operator cannot accidentally adjust a parameter while looking at the wrong page. The display and the physical input are always in sync.
  • Graceful degradation. If no input arrives, the display falls back to the most useful passive information (connection status). The display is never blank or stale.

Idle Timeout Calibration

5 seconds is the threshold for returning to IDLE. This was chosen because: typical mixing gestures (adjusting a zone, tweaking an encoder) complete in 1–3 seconds. A 5s timeout gives comfortable margin for “adjust, pause, re-adjust” workflows without snapping back to connection page mid-thought. Too short (<3s) causes jarring context switches during natural pauses. Too long (>10s) makes the connection page feel unresponsive when the operator actually wants to check connectivity.

Colour System

  • #FFB000 — Amber accent. Headers, active state indicators.
  • #FF4444 — Zone 1 (warm red). Strongest visual weight.
  • #44FF44 — Zone 2 (green). Maximum contrast against Zone 1.
  • #4488FF — Zone 3 (blue). Completes the triad.
  • #FFFFFF — Master brightness. White distinguishes it from zone-specific controls.
  • #22CC44 — Connected / success states.
  • #CC2222 — Disconnected / error states.
  • #0a0a0a — Background. Slight warmth prevents eye fatigue in dark environments.

Implementation Notes

  • State machine: 4 states. Transitions driven by input events from InputManager. No circular references — CONFIRM always returns to the previous non-CONFIRM state.
  • Overlay independence: The overlay has its own 2s timer and renders last (on top). It does not affect state transitions.
  • Memory: Only one page rendered at a time. No off-screen buffers for inactive pages. Rendering cost is constant regardless of how many pages exist.
  • Debounce: Encoder/scroll events reset the idle timer. Rapid input does not cause flickering between states — each event extends the activity window.

Hardware Mapping

  • 4 Scroll Wheels (via PaHub+M5Scroll): Zone 1, Zone 2, Zone 3, Master → triggers ZONE context
  • 2 M5Rotate8 Encoders (8 knobs each): Effect, Speed, EdgeMixer, etc. → triggers ENCODER context
  • AtomS3 Button: Zone toggle, preset save → triggers CONFIRM context
  • AtomS3 LCD: 128×128 ST7789, 0.85″ → renders current context page
